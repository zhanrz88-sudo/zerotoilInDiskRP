## New gap discovered

While the validation was a success for the two intended fixes, the report surfaced a **new gap** on the `get_account()` integration:

```
WARNING: Could not resolve storage tenant for 'dunghnsfailoversrc1':
  Storage account dunghnsfailoversrc1 does not exist in environment Production.
...
Single-side PrepareFailover: searching 1 tenant(s)
No storage tenant resolved -- cannot search XDS logs
```

The TSG falls back to escalation, which is safe — but XDS log search never runs, so we lose the most useful diagnostic signal for `PrepareFailover` stuck.

### Root cause hypothesis

`get_account()` is called with `environment="Production"` (because `tsg_input.environment == "Public"` and we currently map `Public → Production`). The account `dunghnsfailoversrc1` lives in the **PreProd** XStore environment because the alert came from `RSRPPublicPreprodEastUS2`. The lookup tenant_name embeds "Preprod" but our environment mapping does not.

### Proposed follow-up

1. Detect `Preprod` in the RSRP tenant name and map to a PreProd-aware environment string for `get_account()`.
2. Add a unit test that exercises `_extract_input_from_incident()` on a real `RSRPPublicPreprod*` title and asserts the chosen environment is PreProd-compatible.
3. If the lookup still fails in PreProd, gracefully degrade — the dry-run report already shows the rest of the pipeline still produces a useful escalation summary.
